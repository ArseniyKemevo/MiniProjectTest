# Финальный отчёт о разработке проекта «Валютный конвертер»

## 1. Описание пройденного пути по этапам

Разработка приложения была разделена на 6 регламентированных этапов:

1. **Инициация и требования:** Были сформулированы User Stories для трех ключевых ролей (управление курсами, конвертация, просмотр истории) и определены границы MVP. Задачи были зафиксированы на Канбан-доске GitHub Projects.
2. **Проектирование:** Приложение разделено на три независимых слоя (3-Tier Architecture): слой данных (`database.py`), бизнес-логика (`converter.py`) и интерфейс (`main.py`). Базой данных выбрана встроенная СУБД SQLite.
3. **Разработка по TDD:** Сначала создавались модульные тесты в `pytest` для проверки граничных условий и валидации, а затем писался соответствующий код. Было достигнуто **100% покрытие кода тестами** для модулей логики и БД.
4. **Приёмочное тестирование:** На основе требований был составлен ручной чек-лист из 12 сценариев (`TEST_CHECKLIST.md`). Приложение успешно прошло стабилизацию и получило первый релизный тег `v1.0.0`.
5. **Сопровождение и поддержка:** Имитирована обработка 5 обращений от пользователей. Были оперативно устранены критические баги валидации пользовательского ввода (запятые, пробелы-разделители) с предварительным написанием тестов на каждый баг. Выпущены патчи `v1.0.1` и `v1.0.2`.
6. **Ретроспектива:** Проведен финальный анализ проделанной работы.

## 3. Ответы на вопросы ретроспективы

### Что было самым сложным в тестировании?
Самым сложным оказалось обеспечение полной изоляции тестов при работе с базой данных SQLite. Первоначальная идея использовать базу данных в оперативной памяти (`:memory:`) привела к логической ошибке: встроенная БД уничтожалась сразу же после закрытия соединения внутри функции инициализации, из-за чего последующие тесты не находили таблицы `rates` и `history`. 

Проблема была решена созданием кастомной фикстуры `test_db` с использованием встроенного механизма `tmp_path` в `pytest`. Это позволило генерировать уникальный изолированный файл базы данных на диске для каждого отдельного теста и автоматически удалять его по завершении процесса.

### Как изменилось бы приложение, если бы вы сразу знали обо всех багах?
Если бы граничные случаи ввода были известны на этапе проектирования, архитектура функций `validate_amount` и `validate_rate` изначально была бы спроектирована по принципу «либерального приема и строгого сохранения» (Закон Постела). Функции очистки строк (удаление пробелов, конвертация запятых в точки) были бы вынесены в отдельный хелпер-модуль для санитизации данных на самом раннем этапе, что избавило бы от необходимости переписывать регулярные выражения и блоки `try-except` в процессе поддержки.

### Чему вы научились в процессе «поддержки»?
Процесс поддержки наглядно доказал, что реальное поведение пользователей всегда отличается от идеальных представлений разработчика (пользователи активно копируют суммы из банковских приложений с пробелами-разделителями или ставят запятые вместо точек). 

Главный урок поддержки в стиле TDD: **никогда не чинить баг до того, как написан тест, его воспроизводящий**. Написание падающего теста на баг с запятой позволило зафиксировать дефект в коде, а его успешное прохождение после исправления гарантировало, что данный баг больше никогда не вернется (защита от регрессионных ошибок).

## 4. Личная мини-ретроспектива

* **Что удалось:** Успешно реализована строгая трехслойная архитектура. Бизнес-логика полностью отделена от консольного ввода-вывода, что позволило добиться 100% покрытия тестами. База данных защищена составными первичными ключами, исключающими дублирование валютных пар.
* **Что можно улучшить в следующий раз:** В будущем стоит добавить автоматическую загрузку базовых курсов через внешнее API центральных банков в качестве резервного источника данных, а также реализовать полноценный графический интерфейс (GUI) на Tkinter/CustomTkinter для повышения удобства использования.
